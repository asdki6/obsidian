# LLM Wiki 知识库管理运营体系

> **核心命题**：用 LLM 替代人类做知识库的"簿记"工作——总结、交叉引用、更新一致性、标记矛盾。人类负责策展来源、提出问题、深度思考。知识库因此变成**可复利增长的持久化产物**（persistent, compounding artifact），而非每次从零 RAG 检索。

---

## 一、核心理念：Wiki 是一个可复利增长的持久化产物

### 传统 RAG 的局限

| 维度 | RAG 模式 | LLM Wiki 模式 |
|------|----------|---------------|
| 知识状态 | 每次查询从零拼凑 | 知识被编译一次，持续保鲜 |
| 积累性 | 无积累，查询即消失 | 每次摄入和提问都在增长 Wiki |
| 交叉引用 | 每次重新发现 | 已预制在 Wiki 中 |
| 矛盾处理 | 每次重新识别 | 已标记、已对照 |
| 综合归纳 | 每次从碎片拼凑 | 已有演进中的综合论点 |

### 核心隐喻

> **Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。**

你几乎从不亲自写 Wiki——LLM 写和维护全部内容。你的角色是：策展来源、探索方向、提出好问题。LLM 处理所有苦力活：总结、交叉引用、归档、簿记。

## 二、三层架构

```
┌─────────────────────────────────┐
│           Schema（模式）          │  ← 你与 LLM 共同演化
│   告诉 LLM 如何结构化、如何运作    │
├─────────────────────────────────┤
│          Wiki（知识层）           │  ← LLM 全权拥有和编写
│  摘要、实体页、概念页、对比、综合   │
├─────────────────────────────────┤
│       Raw Sources（原始来源）      │  ← 你策展，LLM 只读不写
│   文章、论文、图片、数据文件        │
└─────────────────────────────────┘
```

### Layer 1：Raw Sources（原始来源）
- 你策展的源文件集合（文章、论文、图片、数据）
- **不可变**——LLM 只读不写
- 这是你的**真实来源（source of truth）**

### Layer 2：Wiki（知识层）
- LLM 生成的 markdown 文件目录
- 内容包括：摘要、实体页、概念页、对比、综合论述
- LLM **完全拥有**这一层：创建页面、根据新来源更新、维护交叉引用、保持一致性
- 你读，LLM 写

### Layer 3：Schema（模式文档）
- 一个配置文件（如 `CLAUDE.md` 或 `AGENTS.md`），告诉 LLM：
  - Wiki 的结构约定
  - 页面格式规范
  - 摄入来源、回答查询、维护 Wiki 的工作流
- **这是让 LLM 从通用聊天机器人变成训练有素的 Wiki 维护者的关键**
- 你与 LLM 在使用中不断共同演化这个文档

---

## 三、三大运营操作

### 3.1 Ingest（摄入）

**流程**：
1. 你将新来源放入 Raw Sources
2. LLM 读取来源
3. 与你讨论关键要点
4. LLM 执行以下操作：
   - 在 Wiki 中写摘要页
   - 更新索引
   - 更新相关实体页和概念页（**一个来源可能触及 10-15 个 Wiki 页面**）
   - 在 Log 中追加条目

**摄入模式**：
- **单篇深度模式**（Karpathy 个人偏好）：一次一篇，保持参与——读摘要、检查更新、指导 LLM 侧重方向
- **批量模式**：一次摄入多个来源，减少监督

> 选择适合你风格的流程，并将其写入 Schema 供未来会话使用。

### 3.2 Query（查询）

**流程**：
1. 你对 Wiki 提问
2. LLM 搜索相关页面、阅读、综合回答（附引用）

**输出形式**（取决于问题类型）：
- Markdown 页面
- 对比表
- 幻灯片（Marp）
- 图表（matplotlib）
- Canvas

**关键洞察**：
> **好的回答可以被归档回 Wiki 作为新页面。**
> 一次你问过的对比、分析、发现的关联——这些都是有价值的，不该消失在聊天历史中。这样，你的探索也像被摄入的来源一样在知识库中复利增长。

### 3.3 Lint（健康检查）

定期让 LLM 对 Wiki 做健康检查，检查项包括：

- 页面之间的**矛盾**
- 被新来源**推翻的过时主张**
- **孤立页面**（无入链）
- 被提及但**缺少专属页面的重要概念**
- 缺失的**交叉引用**
- 可以通过网络搜索填补的**数据缺口**

> LLM 擅长建议新的研究问题和寻找新的来源。这保持 Wiki 在增长中保持健康。

## 四、导航基础设施：索引与日志

### index.md（内容导向，目录）
- 列出 Wiki 中每个页面的链接、一句话摘要、可选元数据（日期、来源数量）
- 按类别组织（实体、概念、来源等）
- LLM 每次摄入时更新
- 回答查询时，LLM 先读 Index 找到相关页面，再深入
- **在中等规模（~100 来源，~数百页面）下运作良好**，避免了对 embedding RAG 基础设施的依赖

### log.md（时间导向，日志）
- **仅追加**的操作记录：摄入、查询、Lint
- 每个条目以一致前缀开头（如 `## [2026-04-02] ingest | 文章标题`）
- 使得日志可用简单 Unix 工具解析：`grep "^## \[" log.md | tail -5` 就能看最近 5 条
- 提供 Wiki 演化的时间线，帮助 LLM 了解最近做了什么

## 五、工具生态

| 工具 | 用途 |
|------|------|
| **Obsidian Web Clipper** | 浏览器扩展，将网页文章转为 Markdown |
| **本地图片下载** | Obsidian 设置 → 附件文件夹固定目录（如 `raw/assets/`），绑定快捷键下载图片。让 LLM 可直接查看图片而非依赖可能失效的 URL |
| **Obsidian Graph View** | 查看 Wiki 的形状——哪些页面是枢纽，哪些是孤岛 |
| **Marp** | Markdown 幻灯片格式，从 Wiki 内容直接生成演示 |
| **Dataview** | Obsidian 插件，对页面 frontmatter 运行查询。LLM 添加 YAML 后，可生成动态表格和列表 |
| **qmd** | 本地 Markdown 搜索引擎，混合 BM25/向量搜索 + LLM 重排。有 CLI（LLM 可 shell out）和 MCP Server（LLM 可作原生工具） |
| **Git** | Wiki 就是 Markdown 文件的 Git 仓库，免费获得版本历史、分支、协作能力 |

## 六、为什么这个体系有效

| 人类擅长 | LLM 擅长 |
|----------|----------|
| 阅读和思考 | 总结 |
| 策展来源 | 交叉引用 |
| 提出好问题 | 保持摘要时效 |
| 理解含义 | 标记新旧矛盾 |
| 指导分析方向 | 跨数十页维护一致性 |

> **人类放弃 Wiki 的原因是维护负担的增长快于价值的增长。LLM 不会无聊、不会忘记更新交叉引用、一次可以触及 15 个文件。维护成本趋近于零，Wiki 得以持续存活。**

## 七、适用范围

这个模式可以应用于多种场景：

- **个人知识管理**：目标追踪、健康、心理、自我提升——归档日记条目、文章、播客笔记
- **深度研究**：数周或数月深入一个主题，阅读论文和报告，增量构建有演进论点的综合 Wiki
- **阅读一本书**：逐章归档，构建角色、主题、情节线的页面（类似 Tolkien Gateway 的粉丝 Wiki）
- **商业/团队**：由 Slack 线程、会议记录、项目文档、客户电话驱动的内部 Wiki
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深潜**——任何需要长期积累知识并希望知识被组织而非分散的场景

## 八、思想渊源

与 Vannevar Bush 1945 年提出的 **Memex** 一脉相承——一个私人的、主动策展的知识存储，文档之间的关联路径与文档本身同等珍贵。Bush 未能解决的是"谁来做维护"——LLM 解决了这个问题。
