# 概念_RAG检索增强生成

## 定义

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种结合信息检索与文本生成的技术架构。它将外部知识库文档向量化索引，在用户提问时检索最相关的片段，将其作为上下文注入 LLM，使回答有据可依。

## 核心原理 / 步骤

### 工作流程
1. **索引阶段**：文档 → 分块（chunking）→ 向量嵌入（embedding）→ 存储到向量数据库
2. **检索阶段**：用户提问 → 向量化 → 相似度搜索 → 返回 Top-K 相关片段
3. **生成阶段**：相关片段 + 用户提问 → 拼入 LLM 上下文 → 生成回答（附来源引用）

### RAG vs LLM Wiki 模式
| 维度 | RAG 模式 | LLM Wiki 模式 |
|------|----------|---------------|
| 知识状态 | 每次查询从零检索 | 知识预编译，持续保鲜 |
| 积累性 | 无积累，知识在对话结束后消失 | 持久化产物，可复利增长 |
| 交叉引用 | 每次重新发现 | 已预制在 wiki 页面中 |
| 矛盾处理 | 每次重新识别 | 已标记、已对照 |
| 维护成本 | 索引维护（低） | 页面维护（由 LLM 承担） |
| 适用规模 | 海量文档快速检索 | 精选来源深度积累 |

### 两者不是对立，而是互补
- **RAG**：适合「大海捞针」——在大量文档中快速找到相关信息
- **LLM Wiki**：适合「深度积累」——对筛选过的来源做结构化沉淀
- **理想组合**：RAG 做初筛和搜索，LLM Wiki 做深度加工和持久化

## 应用场景
- **Smart Connections 插件**：Obsidian 中最流行的 RAG 实现
- **企业知识库**：客服系统用 RAG 从产品文档中检索答案
- **学术研究**：从数百篇论文中检索相关段落
- **法律/医疗**：专业文档的精准检索

## 在本知识库中的体现

本 vault 目前采用 **LLM Wiki 模式** 而非 RAG：
- `wiki/index.md` 作为轻量索引（在 ~100 来源规模下足够）
- 当 wiki 增长到数百页面时，可考虑引入 qmd（BM25+向量混合搜索）增强检索能力
